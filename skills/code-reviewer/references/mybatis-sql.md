# MyBatis XML SQL 审查检查清单

## 1. 安全性 (Security)
- [ ] **SQL 注入防范：** 严禁在非必要场景下使用 `${}`。对于表名、列名等动态参数，必须在 Java 层进行严格的白名单校验。
- [ ] **参数绑定：** 优先使用 `#{}` 进行预编译参数绑定，确保 MyBatis 处理 SQL 注入风险。
- [ ] **敏感数据：** 检查 SQL 中是否涉及明文存储的敏感信息（如手机号、身份证），考虑是否需要在数据库层或通过插件进行脱敏处理。

## 2. 性能优化 (Performance)
- [ ] **禁止 SELECT *：** 明确列出所需字段，减少网络带宽消耗及索引覆盖失效。
- [ ] **大分页优化：** 避免在偏移量很大时使用简单 `LIMIT offset, limit`，检查是否使用了覆盖索引或 ID 流转策略。
- [ ] **集合操作：** 使用 `<foreach>` 批量插入/更新时，控制集合大小（通常建议 500-1000 以内），防止生成超长 SQL 导致数据库报错或内存溢出。
- [ ] **模糊查询：** 避免 `LIKE '%xxx%'` 导致的全表扫描，尽可能使用 `LIKE 'xxx%'` 或搜索引擎。
- [ ] **索引利用：** 检查 WHERE 条件、ORDER BY、GROUP BY 中的字段是否建有合适的索引，避免在索引列上使用函数或进行隐式类型转换。

## 3. MyBatis 动态标签规范 (Dynamic SQL)
- [ ] **结构优化：** 优先使用 `<where>`、`<set>`、`<trim>` 标签，自动处理多余的 `AND/OR` 或逗号，避免手写拼接 `1=1`。
- [ ] **条件严谨性：** `<if test="...">` 中的判空逻辑是否完备？（例如：字符串需判断 `!= null and != ''`，集合需判断 `.size() > 0`）。
- [ ] **Choose/When/Otherwise：** 在多分支选择场景下，是否提供了 `otherwise` 兜底逻辑？

## 4. 结果映射与维护性 (Mapping & Maintainability)
- [ ] **ResultMap：** 复杂的关联查询（一对一、一对多）是否定义了清晰的 `ResultMap`？避免过度依赖 `resultType="map"`。
- [ ] **代码复用：** 复杂的字段列表或重复的 Join 逻辑是否提取到了 `<sql id="...">` 片段中？
- [ ] **ID 命名：** XML 中的 `id` 是否与 Mapper 接口中的方法名严格一致？
- [ ] **注释：** 复杂的业务逻辑 SQL 是否配有必要的注释，说明其设计意图或特殊的优化考虑？

## 5. 项目红线约束 (Hard Rules)
- [ ] **[红线/禁止] 严禁在 XML 中编写复杂的业务逻辑：** SQL 应专注于数据存取。禁止在 SQL 中使用过多的存储过程调用、复杂的递归查询或非标准数据库函数，以保证代码的可迁移性。
- [ ] **[强制] 必须配置映射检查：** 确保所有的映射字段在数据库中均有对应。如果字段发生了变更，必须同步更新 XML，严禁遗留未使用的字段映射。
- [ ] **[强制] 事务一致性：** 涉及多表操作的 Mapper 方法，必须在 Service 层标注 `@Transactional`，并在 XML 层面考虑锁竞争情况（如 `FOR UPDATE` 的谨慎使用）。
